IAOS DASHBOARD — BUILD SPEC v1
"What Needs Action" daily view. Companion to Master Architecture Reference V10.1.

PURPOSE
The default landing surface. Answers one question: what needs my attention right now.
Two priority groups, in this order:
  (A) They're waiting on ME  — inbound replies, overdue callbacks, today's callbacks,
                               sent offers pending response.
  (B) Who I should reach cold — the tier+score call queue.
Layout enforces the order: group A always sits above group B so urgency is structural,
not something Brad has to remember.

GUARDRAILS (non-negotiable, same posture as the rest of IAOS)
- GHL is system of record. Dashboard is READ-ONLY except the three scoped writes below.
- Writes allowed:
    1. Notes  -> POST to the contact Notes sub-resource (/contacts/{id}/notes). New note
       per call attempt, never overwrite.
    2. Callback date/time -> a dedicated custom field (see NEW FIELDS).
    3. Attempt marker -> a dedicated custom field (see NEW FIELDS).
- NEVER touches: tags, pipeline stage, offer_ fields, workflows. No Seller-7 / mail-exit.
- All GHL access through the existing shared ghl.ts + ghl-proxy.ts. Token never client-side.
- Clearing a NUMERICAL/DATE custom field uses null, not empty string (GHL ignores "").
- NO MAO calculator surfaced anywhere on this page — no offer math, no "analyze" button.

PAGE LAYOUT (top to bottom)

1. ACTION TILES (row of counts, each click-throughs to its list)
   - Tasks due today
   - Offers to review = contacts with saved offer_ fields but no offer yet sent.
     (Count only. No offer values, no calculator link shown.)
   - Mailers ready this week  (reuse the existing mailer-shared.ts query)
   - New since last login = contacts created after Brad's previous session. Shown as a
     marker/badge, not a full section.

2. WAITING ON ME  (group A — sits above the cold queue)
   Sub-groups appear in the fixed priority order below. Groups hold their rank; the
   overdue-first rule sorts WITHIN a group, never across groups (a longest-waiting
   inbound always outranks an overdue callback, which always outranks a cold lead).
   2.1 Unanswered inbound: contacts with an inbound SMS/email in GHL Conversations with
       no outbound reply after it. HIGHEST priority — top of the entire queue. Reads GHL
       conversations. SORT: OLDEST unanswered reply first (longest-waiting seller on top).
       Rationale: the seller ignored longest is the one closest to giving up — not newest.
   2.2 Callbacks: SORT overdue first (most overdue on top), then callbacks due today by
       scheduled time. overdue = callback_datetime in the past, still not actioned (flag
       late); today = callback_datetime is today.
   2.3 Offers awaiting seller response: offer has been SENT (not just calculated) and is in
       the waiting window. Only include if the offer/Seller workflow exposes a "sent/awaiting"
       signal that can be read WITHOUT writing anything. If no clean read signal exists, mark
       this sub-section DEFERRED rather than inferring it — do not guess state.

3. CALL QUEUE  (group B — cold outreach, below group A)
   Contacts who have NOT replied. Sits entirely below group A.
   - Sort: tier + score = the mailer priority (Hot -> Warm -> Low, score DESC within each).
     Any overdue item within this group sorts to the top of its tier (overdue-first rule).
   - FILTER: has a valid phone. BUCKET-AGNOSTIC. Low-bucket contacts WITH a phone ARE in
     the queue, ranked below Hot/Warm. Only address-only contacts (no phone) are excluded
     from calling — they still get mailed elsewhere. Do NOT filter the queue by bucket.
   - Row shows: name, phone, tier, score, address, last-contact indicator.
   - CLICK-TO-CALL button per row (see CALLING).
   - NOTES field per row: autosave on blur -> writes a NEW timestamped GHL note. A saved
     note is the "attempted" signal (sets attempt marker + timestamp).
   - After a note saves, the row GREYS OUT / drops to bottom. (Resurface rule = OPEN ITEM.)
   - SCHEDULE-CALLBACK control: calendar + time popup for fast entry; writes callback_datetime.

4. PIPELINE HEALTH STRIP  (small, single row, must NOT dominate the page)
   - Contact counts per bucket + per pipeline stage. Read-only.
   - "Deals in active negotiation" (contacts in working pipeline stages) folds in HERE,
     not its own section.

CALLING — Option A + Option B, investor-selectable
Telephony is GHL/Lead Connector (Twilio underneath). Number +1 972-954-8586 is live, A2P
verified, SHAKEN/STIR + CNAM + Voice Integrity all approved. Both call modes are a GHL
setting, NOT two code paths — the click-to-call button is identical in both.
   - Option A — Browser softphone (WebRTC): call rings in the browser, uses computer
     mic/speakers. USB headset recommended (not required).
   - Option B — Forward-to-phone: GHL rings Brad's cell/desk first, then bridges to the
     seller. Set via the number's Forwarding Number field.
   - IMPLEMENTATION: one custom value  call_mode  (values: softphone | forward) read
     per-account at load, same pattern as mailer_digest_recipient (fixed ID in code,
     per-account value). In forward mode, read a second custom value  call_forward_number
     for the destination. Button behavior does not branch on mode — mode is a GHL/config
     concern. This makes both modes available to future SaaS clients by config, zero rebuild.
   - NO MAKE REQUIRED. GHL logs calls natively. Make is phase-2 ONLY if a call OUTCOME
     should auto-trigger a workflow (e.g. no-answer -> auto next touch). Out of scope for v1.
