IAOS DASHBOARD — BUILD SPEC v2
"What Needs Action" daily view. Companion to Master Architecture Reference V10.2. (v2: Phase-2 review — layout reorder, Call Queue renamed Lead Queue, resurface/grey-out rules, escalation, 15-row cap.)

PURPOSE
The default landing surface. Answers one question: what needs my attention right now.
Two priority groups, in this order:
  (A) They're waiting on ME  — inbound replies, overdue callbacks, today's callbacks,
                               sent offers pending response.
  (B) Who I should reach cold — the tier+score call queue.
Layout enforces the order: group A always sits above group B so urgency is structural,
not something Brad has to remember.

GUARDRAILS (non-negotiable, same posture as the rest of IAOS)
- GHL is system of record. Dashboard is READ-ONLY except the three scoped writes below.
- Writes allowed:
    1. Notes  -> POST to the contact Notes sub-resource (/contacts/{id}/notes). New note
       per call attempt, never overwrite.
    2. Callback date/time -> a dedicated custom field (see NEW FIELDS).
    3. Attempt marker -> a dedicated custom field (see NEW FIELDS).
- NEVER touches: tags, pipeline stage, offer_ fields, workflows. No Seller-7 / mail-exit.
- All GHL access through the existing shared ghl.ts + ghl-proxy.ts. Token never client-side.
- Clearing a NUMERICAL/DATE custom field uses null, not empty string (GHL ignores "").
- NO MAO calculator surfaced anywhere on this page — no offer math, no "analyze" button.

PAGE LAYOUT (top to bottom) — REVISED v2 per Brad's Phase-2 review

Ordering principle: the most-missed content goes highest, because nobody scrolls to
find it. Action tiles and Pipeline Health are glanceable status that must be "in your
face"; Waiting on Me is the priority work; Lead Queue is the long cold list at the bottom.

1. ACTION TILES (row of counts, read-only)
   - Tasks due today
   - Offers to review = contacts with saved offer_ fields but no offer yet sent.
     (Count only. No offer values, no calculator link shown.)
   - Mailers ready this week (reuse mailer-shared.ts)
   - New since last login = badge, not a section.

2. PIPELINE HEALTH STRIP (small, single row)
   MOVED UP (was below the queue in v1). Rationale: it's glanceable status nobody would
   scroll for, so it must sit high where it's seen. Contact counts per bucket + per
   pipeline stage; "deals in active negotiation" folds in here. Read-only. Must stay
   compact — one row, does not dominate.

3. WAITING ON ME (priority work — anyone who has engaged)
   A contact who has responded in ANY way (inbound SMS/email, seller-form submission,
   and later a booked appointment) is REMOVED from the Lead Queue and escalated here.
   They are never a cold call once they have engaged. Sub-groups in fixed priority order;
   overdue-first sorts WITHIN a group, never across groups.
   3.1 Unanswered inbound: inbound SMS/email in GHL Conversations with no outbound reply
       since. SORT: OLDEST unanswered reply first (longest-waiting seller on top). LIVE.
   3.2 Callbacks: overdue first (most overdue on top), then today by scheduled time.
       Reads callback_datetime.
   3.3 Offers awaiting seller response: offer SENT and in the waiting window (opportunity
       in "Seller Offer Sent" stage carrying offer-made tag — no-write read signal). LIVE.
   3.4 Appointments (FUTURE): booked calendar appointments. Deferred until Calendars phase.

4. LEAD QUEUE (renamed from "Call Queue" — cold outreach, the long list at the bottom)
   Contacts who have NOT engaged. Everything about the cold-call worklist lives here.
   VISIBLE CAP: show ~15 rows; the remainder are reached by SCROLL within the section
   (RESURFACE_VISIBLE_ROWS = 15, tunable). Do not render all 39+ at once.

   SORT (three bands, top to bottom):
     BAND 1 — Attempted-but-no-response, OLDEST last-attempt first. These are contacts
       you have called at least once (last_call_attempt set) who have NOT engaged. The
       one you called longest ago rises highest — persistence on touched leads beats
       spraying brand-new ones. Sort ascending by last_call_attempt.
     BAND 2 — Never-attempted, by tier + score (Hot -> Warm -> Low, score DESC within).
       This is the original mailer-priority sort. Overdue-within-tier bubbles to top of
       its tier.
     BAND 3 — Freshly-attempted (GREYED), at the very bottom. See GREY-OUT below.

   FILTER: has a valid phone. BUCKET-AGNOSTIC. Low-bucket contacts WITH a phone are IN
   the queue (below Hot/Warm). Only address-only (no phone) are excluded from the queue.

   ROW: name, phone, tier, score, address, last-contact indicator, Call button, Notes.

   GREY-OUT / ATTEMPT STATE (Phase 2 core):
   - MECHANISM (from source): a fresh last_call_attempt (< RESURFACE_HOURS) greys a row and
     drops it to BAND 3. Nothing else greys; notes are NOT read by the grey computation.
   - CONVENTION: a note is how a human marks an attempt. Saving any non-empty note (1 character
     is enough, e.g. "." or "x") writes a NEW timestamped GHL note AND sets last_call_attempt to
     now, in that gated pairing. The note is the human-readable record; last_call_attempt is what
     greys. A note without setLastCallAttempt would NOT grey; setLastCallAttempt without a note
     would grey with no record of why.
   - The Call button is INDEPENDENT and DIAL-ONLY. Clicking Call does NOT set last_call_attempt
     and does NOT grey the row. A call with no note leaves the row UN-GREYED on purpose — the
     visual nudge that the call has not been logged yet. Rationale (Brad): never allow a call
     with no note in the system; and allow marking attempted WITHOUT calling (a note like
     "bad number" or "try Thursday" writes last_call_attempt and greys the row without a dial).
   - Opening a contact and even clicking Call, with NO note saved, does NOT grey it.
   - NOTES FIELD AUTO-FOCUS: when a contact opens, the cursor auto-lands in the notes field
     (no click needed) so Brad can type immediately. Cursor stays there until he clicks out.
     Purpose: remove all friction from note entry, since the note is mandatory for an attempt
     to register. Autosave on blur.
   - CALL BUTTON MUST NOT STEAL FOCUS: clicking Call does NOT move the cursor out of the notes
     field (build the button to not grab focus - mousedown preventDefault or tabIndex=-1). Brad
     can be mid-note, click Call, and keep typing. Only clicking another input or empty space
     blurs the notes field. Consequence (consistent): since Call does not blur, clicking Call
     does NOT fire the autosave and does NOT create the attempt - the note saves only when Brad
     actually clicks away. This matches the note→setLastCallAttempt pairing (MECHANISM above) exactly.
   - On attempt, the row GREYS and drops to BAND 3 (bottom). Greyed rows stay VISIBLE at
     the bottom (not hidden), reachable by scroll.
   - Grey is INFORMATIONAL, NOT a lockout. It only signals "you already tried this one."
     The row stays fully clickable; Brad can reopen and call again inside the window
     anytime. Grey never prevents a callback.
   - Clicking a greyed row OPENS the contact (edit notes / click-to-call again). It does
     NOT un-grey. Only the timer un-greys it.
   - 12-HOUR AUTO-RESET (RESURFACE_HOURS = 12, flat, tunable in ONE place): when
     now - last_call_attempt >= 12h, the row auto-un-greys and rejoins the active sort
     as if no attempt was made for ranking purposes — BUT all notes are KEPT (notes are
     permanent GHL notes; only the visual attempt-state expires). After un-grey it re-enters
     BAND 1 (attempted-but-no-response, oldest-first) since last_call_attempt still exists
     and is now the oldest.
   - Overnight everyone attempted resets, so each morning starts with a full active queue.

   Escalation out (see WAITING ON ME §3): the instant a contact has an inbound signal,
   they leave the Lead Queue entirely and appear under Waiting on Me. Phase 2 covers the
   text/email signal (live via ghl-conversations). Seller-form and appointment signals are
   added as those reads come online (form: confirm signal; appointment: Calendars phase).

CALLING — REVISED (Build 2B): GHL's public API cannot trigger an outbound call
Researched, not guessed: GHL's Conversations/Calls API can only LOG a call that already
happened (or pull recordings/transcripts of past calls) — there is no endpoint to
programmatically dial through the account's Twilio-backed number. An open GHL feature
request for exactly this ("API endpoint for calls") is unresolved on their own roadmap.
This breaks the original plan below of one in-app button branching on call_mode.

OPTION 1 — SHIPPED ✅: click-to-call opens the contact's page inside GHL's own app
(https://app.gohighlevel.com/v2/location/{locationId}/contacts/detail/{contactId}) in a
new tab via window.open — no GHL API call at all. Brad clicks GHL's native call icon
there; GHL's own dialer applies the Number's own softphone/forward config, using the
verified +1 972-954-8586 number (A2P, SHAKEN/STIR, CNAM, Voice Integrity all approved).
Opening the tab is independent of the attempt system — it does NOT set last_call_attempt
or grey a row (grey keys off a fresh last_call_attempt, which only the note→setLastCallAttempt
pairing writes; unchanged from Build 2A). Verified: still does not steal focus from the notes
field (same tabIndex=-1 + mousedown-preventDefault guard).

call_mode / call_forward_number custom values: DEAD for call-routing purposes under
Option 1 — GHL's dialer reads its own Number config, not these custom values, so there's
nothing for this app to branch on. Per Brad's call: left in place in GHL, unused by any
code, no UI built around them. Not deleted — future options (below) could still use them.

OPTION 2 — Forward-to-phone: unchanged as a GHL-side account setting (Number's Forwarding
Number field); no app code involved either way, since Option 1 hands off to GHL regardless
of which mode the account is in.

OPTION 3 (DEFERRED) — Marketplace OAuth app + embedded dialer SDK: GHL marketplace apps
can embed richer UI (custom pages, possibly a dialer widget) via OAuth, a much bigger lift
than the current Private Integration Token setup. Only worth it if GHL-tab-hopping (Option
1) becomes a real bottleneck at higher dialing volume. Option 1 does NOT block adopting
this later — it's an additive upgrade path, not a dead end.

NO MAKE REQUIRED. GHL logs calls natively. Make is phase-2 ONLY if a call OUTCOME
should auto-trigger a workflow (e.g. no-answer -> auto next touch). Out of scope for v1.

OPEN ITEMS (resolve before or during the phase that needs them; do NOT guess)

     lockout. 12h flat auto-reset (RESURFACE_HOURS=12, tunable). Click-to-open (not un-grey);
     only the timer un-greys. Sort: oldest-attempt band on top, never-attempted by tier+score,
     freshly-attempted greyed at bottom. Notes always kept. See LEAD QUEUE section.
  2. Offers-awaiting-response: confirm whether a no-write read signal for "offer sent"
     exists. If not, stays DEFERRED. (Needed Phase 3.)
  3. Live test call: first outbound from +1 972-954-8586 — walk through together. (Phase 4.)
  4. last_call_attempt date/time precision: ACCEPTED LIMITATION (Build 2A live verification,
     Neelima Bale test note). GHL's last_call_attempt field is typed DATE, not DATETIME — it
     silently truncates the time-of-day on write, storing e.g. "2026-07-13" even though the
     app sends a full ISO timestamp. The 12h grey/un-grey window is exact within a single
     browser session (an in-memory timestamp override carries the real time), but after a
     reload the app re-derives last_call_attempt from GHL's date-only value, losing hour-level
     precision across sessions. Accepted as-is: dialing stays inside 8am-8pm, so the 12h window
     never straddles a date boundary in practice. Revisit only if calling hours ever expand
     past midnight-crossing territory.
