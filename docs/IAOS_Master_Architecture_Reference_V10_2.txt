1. Product Identity

Standalone branded software on top of GoHighLevel — not a reseller/whitelabel. Reads and writes GHL data via API through IAOS's own interface.
Dual purpose: (a) Brad's own wholesaling tool; (b) sellable SaaS for other wholesalers (FMTM).
Default rule: when ambiguous, choose the client-facing option. Brad is "client #1."
PRODUCT DIRECTION — FULL OPERATIONAL FRONT-END (locked): IAOS is the complete day-to-day workspace for lead work. Clients do ALL contact/lead operations in the app — dashboard, conversations, calendars, contacts, opportunities — and touch GHL directly ONLY for configuration or repair, never for working leads. Coverage is built out surface by surface (see §2a Coverage Roadmap); each surface is designed client-first for someone working leads, NOT as a clone of GHL's own layout. GHL remains system of record throughout — this is a front-end over GHL, not a fork of it.

2. Three-Layer Model

Layer 1 — System of Record: GoHighLevel. Contacts, custom fields, opportunities, stages, workflows, comms, compliance. Never duplicated.
Layer 2 — Orchestration + Compute: Make.com + Netlify functions. Make = cross-system glue. Netlify = pure compute (scoring, MAO, Mailers digest).
Layer 3 — Presentation: IAOS frontend shell. Standalone, GHL-mirrored, IAOS-branded. LIVE at app.investorautomationos.com.

11. Known GHL Constraints & API Lessons

Opportunity list view: can't show custom scores. Contacts list: can. Kanban: sorts by stage only. List customizations: browser-specific, not backend-saved.
GHL branches do NOT rejoin/merge. Consequence: a per-touch If/Else pattern dead-ends every branch except the one carrying the sequence — broke the first two Hot builds. Tier cadences built as ONE linear chain gated by a SINGLE If/Else at the top (§14b). Any tail action that must run on more than one branch has to be DUPLICATED per branch.
GHL's Tasks API is CONTACT-SCOPED ONLY (GET /contacts/{id}/tasks) — there is no location-wide "list all tasks" endpoint. Mailers gathers all mailer tasks by fetching all contacts, then fanning out one tasks-fetch per contact with bounded concurrency (5-way pool) rather than the fully-serial pacing used for the single-endpoint contacts/opportunities pagers — a serial fan-out risks Netlify's ~10s function timeout as contact count grows (§14f).

API LESSONS (from Pipeline + MAO + Mailers builds):
- Opportunity custom fields come back under fieldValueNumber / fieldValueDate keys for NUMERICAL/DATE types — NOT the value / fieldValue keys much of the repo assumed. Any function reading opportunity custom fields must check BOTH shapes (cfNum/cfText helpers do this). The existing mao-webhook.ts predates this fix.
- WRITING offer_ fields with a plain "field_value" key works for the Contact + Opportunity custom-field PUT used by Save Offer (verified live in Phase 6 — the numbers landed correctly on both records).
- Clearing a NUMERICAL/DATE custom field: empty-string is SILENTLY IGNORED by GHL; use null to actually clear it.
- Custom-field FOLDERS can be created via API: POST /locations/{id}/customFields with documentType:"folder", per model (contact / opportunity separately).
- Contact search native query param matches name/email/phone ONLY — NOT address. For address search, fetch contacts (listAll, paginated) and filter name+address client-side. (Search UI needs ≥2 correct characters before it fires — a debounce, not a bug.)
- /opportunities/search does NOT accept a contactId filter (returns 422). To find a contact's opportunities, reuse ghl-opportunities.ts (list by pipeline) and match contactId client-side.
- Moving an opportunity stage via the standard opportunity-update API DOES fire GHL's native workflow triggers (verified: manual move to "Seller Offer Sent" fired Seller 7). Never write stage via a path that bypasses triggers. CONVERSELY: a custom-fields-ONLY PUT (no pipelineStageId, no status) does NOT touch the stage and cannot fire stage triggers — this is exactly how Save Offer stays inert (§14e).
- Contact tasks have a dedicated completion sub-resource: PUT /contacts/{contactId}/tasks/{taskId}/completed with body {completed:true} — flips ONLY the completed flag, structurally cannot touch title/dueDate/assignedTo (§14f).
- companyName is a NATIVE contact field (like address1/city/state), not a custom field — confirmed live. Currently null on all 40 test contacts (individuals, not businesses), so the Mailers business-flagged list is legitimately empty until real business-owner leads appear.
- Local dev: netlify dev resolves .env relative to whatever directory it's run from (its --dir/base, not necessarily repo root) — for the app site (base "app"), the token must be in app/.env, not just the root .env. Both are gitignored. Running `netlify dev` from inside app/ without an explicit --functions override can also incorrectly inherit a DIFFERENT linked site's function directory if a parent .netlify/state.json exists (it picked up the marketing site's ./netlify/functions instead of app/netlify/functions) — pass --functions=netlify/functions explicitly when running from app/ to force the right directory. FIXED THIS SESSION (V9): the recurring token-extraction failures were caused by a two-line .env (GHL_PRIVATE_API_KEY + a duplicate GHL_API_TOKEN) combined with a non-anchored sed pattern that let the second line through. Root .env is a SINGLE clean line (GHL_PRIVATE_API_KEY only).
