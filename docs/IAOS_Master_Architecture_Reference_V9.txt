11. Known GHL Constraints & API Lessons

Opportunity list view: can't show custom scores. Contacts list: can. Kanban: sorts by stage only. List customizations: browser-specific, not backend-saved.
GHL branches do NOT rejoin/merge. Consequence: a per-touch If/Else pattern dead-ends every branch except the one carrying the sequence — broke the first two Hot builds. Tier cadences built as ONE linear chain gated by a SINGLE If/Else at the top (§14b). Any tail action that must run on more than one branch has to be DUPLICATED per branch.

API LESSONS (from Pipeline + MAO builds):
- Opportunity custom fields come back under fieldValueNumber / fieldValueDate keys for NUMERICAL/DATE types — NOT the value / fieldValue keys much of the repo assumed. Any function reading opportunity custom fields must check BOTH shapes (cfNum/cfText helpers do this). The existing mao-webhook.ts predates this fix.
- WRITING offer_ fields with a plain "field_value" key works for the Contact + Opportunity custom-field PUT used by Save Offer (verified live in Phase 6 — the numbers landed correctly on both records).
- Clearing a NUMERICAL/DATE custom field: empty-string is SILENTLY IGNORED by GHL; use null to actually clear it.
- Custom-field FOLDERS can be created via API: POST /locations/{id}/customFields with documentType:"folder", per model (contact / opportunity separately).
- Contact search native query param matches name/email/phone ONLY — NOT address. For address search, fetch contacts (listAll, paginated) and filter name+address client-side. (Search UI needs ≥2 correct characters before it fires — a debounce, not a bug.)
- /opportunities/search does NOT accept a contactId filter (returns 422). To find a contact's opportunities, reuse ghl-opportunities.ts (list by pipeline) and match contactId client-side.
- Moving an opportunity stage via the standard opportunity-update API DOES fire GHL's native workflow triggers (verified: manual move to "Seller Offer Sent" fired Seller 7). Never write stage via a path that bypasses triggers. CONVERSELY: a custom-fields-ONLY PUT (no pipelineStageId, no status) does NOT touch the stage and cannot fire stage triggers — this is exactly how Save Offer stays inert (§14e).
- Local dev: netlify dev only picks up the token from the root .env FILE content (shell exports alone don't work). app/.env and .env are gitignored. FIXED THIS SESSION: the recurring token-extraction failures were caused by a two-line .env (GHL_PRIVATE_API_KEY + a duplicate GHL_API_TOKEN) combined with a non-anchored sed pattern that let the second line through. .env is now a SINGLE clean line (GHL_PRIVATE_API_KEY only), no BOM, no second line. Keep it that way.

14b. Tier Mail-Cadence Workflows — BUILT & PUBLISHED ✅

Three GHL workflows, each triggered by its bucket tag. Separate from Seller 0 (its own trigger-less digital engine). No coupling.
ARCHITECTURE (all three): ONE linear chain gated by a SINGLE If/Else at the top (see §11). Trigger: Contact Tag added. If/Else "Mail Automation OFF": custom value mail_automation_enabled Is off → OFF branch carries the whole sequence; None branch empty (toggle-ON/vendor path, §14c). OFF branch: Create Task → Wait → ... → demotion tail. Each Create Task: assigned to Brad, due 0 days, 8:00 AM (cloned tasks drop due dates — re-verify).
Live workflow names: "Sequence - Hot Tag Mail", "Sequence - Warm Tag Mail", "Sequence - Low Tag Mail".
HOT (tag hot): 8 touches, every 2 weeks (~4mo), alternating PRIMARY/POSTCARD. Tail: Remove hot → Add warm. PUBLISHED.
WARM (tag warm): 4 touches, monthly (~4mo), alternating. Tail: Remove warm → Add low. PUBLISHED.
LOW (tag low): 4 touches, every 2 months (~8mo), POSTCARD only. Tail: Remove low ONLY (exits mail; stays in Seller 0 unless moved to LTN). PUBLISHED.
Demotion chain: Hot (~4mo) → Warm (~4mo) → Low (~8mo) → out = ~16 months if never engaged. Time-based, in the workflows; scoring function never moves tiers.
Task-based, not slot-tags: each touch creates a transient TASK (the manual "who to mail" list). Contact only ever carries its one tier tag.
Settings: re-entry OFF (re-fired tier tag must not restart a cadence). Skip-weekends OFF on tasks.
Recurring GHL AI builder issues: refuses clean linear chains (build "bones only, no If/Else" first, insert gate manually); off-branch tasks build empty; due dates don't stick; steps sequence out of order; editing a tangled workflow throws network errors (rebuild from empty). Verify after every AI build.

MAO MODEL — buyer-profit-first:
  MAO (Seller Offer) = ARV − Rehab − Selling/Holding − Target Buyer Profit − Assignment Fee
  Assignment Fee = MAX($5,000 floor [adjustable], target fee). Target Buyer Profit = $ amount OR % of ARV.
  Fee is subtracted exactly ONCE (inside MAO). No double-count.
  Verified test: ARV 300k / rehab 50k / selling 30k / buyer profit 40k / fee 10k → MAO 170k.
8 DISPLAYED METRICS: Seller Offer (MAO), Assignment Fee, Buyer Purchase Price (= Seller Offer + Assignment Fee), Buyer Expected Profit, Buyer ROI (= Profit / total project cost = offer+fee+rehab+selling/holding+financing), Profit Margin (= Profit / ARV), Wholesaler Spread % (= Fee / Buyer Purchase Price), Assignment % of Contract (= Fee / Seller Offer). Plus 3 green/red reasonableness checks: fee ≥ $5k; fee ≤ ~25% of buyer profit; buyer meets target return. Margin GREEN positive / RED negative.
SECTION B — FLIP ANALYSIS (FMTM formulas, exact): purchase_closing = offer×2% (editable); holding = (utilities_mo + insurance_yr/12 + taxes_yr/12) × months; selling = ARV×5% + ARV×1.23%; all_in = offer+rehab+closing. CASH: profit = ARV−selling−all_in; roi = profit/all_in. HARD MONEY: total_borrowed = min(all_in, LTV×ARV); monthly = borrowed×rate/12; total_money_cost = monthly×months + addl_fees + borrowed×points%; profit = cash_profit − total_money_cost. PRIVATE MONEY: standard PMT amortization (interest-only toggle supported). "Other/Misc costs" catch-all line feeds margin + project cost.
REPAIR ESTIMATOR (shared, feeds both sections): quick single-number mode OR expandable granular bid sheet (qty × unit cost, category subtotals, +10% fudge). Expand affordance is obvious but NOT labeled "go granular."

offer_ CUSTOM FIELDS: 14 fields created on BOTH Contact and Opportunity, grouped in an "Offer" folder. Names: offer_price, offer_mao, offer_wholesale_fee, offer_repair_total, offer_margin, offer_arv, offer_date. (Money fields typed NUMERICAL — GHL has no "currency" type; offer_date is DATE.)
